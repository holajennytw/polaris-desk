## 背景與問題

同業比較頁引用追蹤器（ctx-panel）點入引用後，`cite-label` 只顯示原始 `source_id`（如 `chunk-2330-2026Q1-abc`）或只有年份，遠不如研究助理頁引用面板清晰。

研究助理頁能顯示「台積電_2026Q1_法說會逐字稿」是因為後端 `POST /research` 回傳的 citation 裡已有 `title`、`doc_type`、`fiscal_period` 等欄位（PR #25 交付的三路檢索 metadata）。

同業比較頁 `POST /peer-compare` 的 citation 結構至今只有：
```typescript
{ src: string; page: string }
```

前端 `peerCitations()` 只能把 `citation.src` 直接當 `label` 顯示，導致 ctx-panel 資訊匱乏。

---

## 根本原因

`source_id` → 中文名稱的轉換屬於 BQ 資料層知識（ontology JOIN）：

| BQ 表 | 中文欄位 | 用途 |
|-------|---------|------|
| `r6_disclosure_event` | `event_type_name` | 事件類型中文名（法說會、重大訊息、月營收公告…） |
| `r6_financial_metric` | `metric_name` | 財務指標中文名（月營收、毛利率、EPS…） |
| `company_dim` | `company_name` | 公司中文名（台積電、聯發科…） |
| `v_chunk_semantic` | `event_type_name`（已 JOIN） | chunk 層查詢已有中文事件名 |
| `v_financial_metrics_semantic` | `metric_name`（已 JOIN） | 財務指標查詢已有中文指標名 |

前端不知道 `source_id` 的格式規則，也不應該解析它（違反前後端契約）。**此轉換必須由後端在 `POST /peer-compare` workflow 內完成。**

## 需求：`POST /peer-compare` Response 新增 citation metadata

### 現有 citation 結構（待更新）

```typescript
// frontend/src/types/api.ts
interface PeerKpiRaw {
  label: string;
  a: { v: string; citations: Array<{ src: string; page: string }> };
  b: { v: string; citations: Array<{ src: string; page: string }> };
  diff: string;
  better: "a" | "b" | "tie";
}
```

### 新增欄位（R3 須補齊）

每個 citation 物件新增兩個選填欄位：

```typescript
interface PeerCitationRaw {
  src:        string;    // source_id（不變）
  page:       string;    // 頁碼（不變）
  label?:     string;    // NEW：人類可讀的文件標題（中文）
  kind_label?: string;   // NEW：文件類型中文名
}
```

`calls` 的 cite 欄也需要補充（目前只有 `cite: string`）：

```typescript
interface CallSideRaw {
  stance: string;
  tone:   string;
  quote:  string;
  cite:   string;        // source_id（不變）
  cite_label?:     string;   // NEW：人類可讀的來源名稱
  cite_kind_label?: string;  // NEW：文件類型中文名
}
```

---

## 各欄位說明與來源

| 欄位 | 說明 | 來源 BQ 表／欄位 | 範例值 |
|------|------|----------------|--------|
| `label` | 文件標題，對應研究助理 `GET /chunk/{id}` 的 `title` | `v_chunk_semantic.source_name` 或 `company_name + fiscal_period + event_type_name` 組合 | `"台積電 2026Q1 法說會逐字稿"` |
| `kind_label` | 文件類型中文名 | `r6_disclosure_event.event_type_name` 或 `v_chunk_semantic.event_type_name` | `"法說會"` / `"月營收公告"` / `"重大訊息"` |
| `cite_label` | calls 法說引用的來源標題 | 同上 | `"台積電 2026Q1 法說會"` |
| `cite_kind_label` | calls 引用的文件類型 | 同上 | `"法說會"` |

> ⚠️ `r6_compliance_term.term_name` 是合規禁詞，**不是**引用標籤的來源，請勿混用。

## 後端實作建議

### 方案 A（推薦）：workflow 內 JOIN 後直接填入

在 peer-compare workflow 的 retrieval 步驟完成後，對每個 citation 的 `source_id`：

1. 查 `v_chunk_semantic` WHERE `source_id = ?`，取 `event_type_name`、`source_name`（或拼出 title）
2. 若 source_id 對應財務指標，查 `v_financial_metrics_semantic`，取 `metric_name`
3. 組成 `label`（優先用 `source_name`；fallback 用 `company_name + fiscal_period + event_type_name`）
4. 組成 `kind_label`（取 `event_type_name`；財務指標用 `"財務指標"`）

### 方案 B：單獨 batch endpoint（不推薦）

前端收到 peer-compare 回應後，把所有 source_id 送 `POST /chunk-labels`，再合併。額外 round-trip，增加前端複雜度，不建議。

## 前端對應動作（R3 交付後）

R7 前端改動最小，主要在 `frontend/src/lib/peer-result.ts`：

```typescript
// peerCitations() 裡 add() 改為優先使用後端提供的 label
kpi.a.citations.forEach((citation) => {
  add({
    sourceId: citation.src,
    label: citation.label ?? citation.src,            // 使用後端 label
    detail: citation.kind_label ?? kpi.label,         // 使用後端 kind_label
    snippet: [kpi.label, kpi.a.v, citation.page].filter(Boolean).join("　"),
  });
});
```

`peer/page.tsx` 不需改動；ctx-panel 的 `cite-label` 元素已存在，只是目前顯示 raw source_id。

## 驗收標準

- [ ] 同業比較引用面板 `cite-label` 顯示中文文件名（如「台積電 2026Q1 法說會逐字稿」），而非 source_id
- [ ] 財務指標類型引用顯示「財務指標」或對應的 `metric_name`（如「毛利率」）
- [ ] 法說引用顯示「法說會」
- [ ] 後端找不到 label 時，欄位可省略（前端 fallback 使用 source_id）
- [ ] 現有 `POST /peer-compare` Request 格式不變（此為 Response-only 變更）
