# 前端待修：對話紀錄點「同業比較」紀錄 → 內容空白

**負責人**：R7（前端）
**開立日期**：2026-06-27
**分類**：純前端 bug（後端不需改）
**優先級**：P1（功能壞掉，使用者點了沒反應）

---

## 1. 問題現象

在「對話紀錄」頁點某一筆**同業比較（peer）**的紀錄後，跳轉到的 `/peer` 頁面是**空白**的（只剩空狀態，沒有還原當初的比較結果）。

> 研究助理（research）的紀錄點進去是正常的——只有 peer 壞掉。

---

## 2. 根因（已定位，不是後端問題）

後端是對的：

- `GET /history/{id}`（`src/polaris/api.py` `get_history_one`）會回傳整包紀錄，**含 `result`**。
- `frontend/src/lib/api.ts` 的 `historyOne()` 也正確回 `{ query, page, result }`。

問題在前端跳轉後的「還原」這一段：

`frontend/src/app/(dashboard)/history/page.tsx` 的 `handleItemClick`（約 63–73 行）會：

```ts
const full = await api.historyOne(item.id);
if (full?.result) {
  sessionStorage.setItem("polaris_restore", JSON.stringify({
    id: item.id, query: full.query, page: full.page, result: full.result, time: item.time,
  }));
  router.push(`/${full.page}?historyId=${encodeURIComponent(item.id)}`);
  return;
}
```

也就是說：**把結果存進 `sessionStorage["polaris_restore"]`，再帶 `?historyId=` 跳轉**。

- `research/page.tsx` **有**對應的「讀 sessionStorage 還原」邏輯 → 正常。
- `peer/page.tsx` **完全沒有** `useSearchParams` / `historyId` / `polaris_restore` 的處理 → 結果永遠沒被讀回來 → 空白。

## 3. 參考實作（research 頁已經這樣做了，照抄即可）

`frontend/src/app/(dashboard)/research/page.tsx`：

**(a) 還原 effect（約 113–131 行）**

```ts
// B 級還原：從 history 頁點進來時，讀 sessionStorage 直接復原結果
useEffect(() => {
  const historyId = searchParams.get("historyId");
  if (!historyId) return;
  try {
    const raw = sessionStorage.getItem("polaris_restore");
    if (!raw) return;
    const saved = JSON.parse(raw);
    if (saved.id !== historyId) return;
    setQuery(saved.query ?? "");
    setHasQueried(true);
    setPhase("done");
    setProgress(100);
    setRestoredData(saved.result);
    setRestoredAt(saved.time ?? null);
    sessionStorage.removeItem("polaris_restore");
  } catch {}
  // eslint-disable-next-line react-hooks/exhaustive-deps
}, []);
```

**(b) Suspense 包裹（約 557–564 行）** —— `useSearchParams()` 一定要包在 `<Suspense>` 內，否則 `next build` 靜態匯出 `/peer` 會 bail-out：

```tsx
// useSearchParams() 需包在 Suspense 內，否則 next build 靜態匯出會 bail-out。
export default function ResearchPage() {
  return (
    <Suspense fallback={null}>
      <ResearchPageInner />

  );

```

## 4. 在 `peer/page.tsx` 要做的修改

檔案：`frontend/src/app/(dashboard)/peer/page.tsx`

1. **import**
   - 從 `react` 多 import `Suspense`。
   - 從 `next/navigation` import `useSearchParams`。

2. **拆元件 + Suspense 包裹**
   - 把現在的 `export default function PeerPage()` 改名成 `function PeerPageInner()`。
   - 新增：
     ```tsx
     export default function PeerPage() {
       return (
         <Suspense fallback={null}>
           <PeerPageInner />
         </Suspense>
       );
     }
     ```

3. **在 `PeerPageInner` 內加還原 effect**（掛載時跑一次）
   - peer 頁的結果是存在 `peerResult`（`PeerCompareResult`），存進 sessionStorage 的 `saved.result` 形狀一致，可直接 `setPeerResult(saved.result)`。
   - 同時要還原**選到的兩家公司**（讓標題與插槽顯示正確）：從 `saved.result.a_ticker` / `b_ticker` 還原。

   ```ts
   const searchParams = useSearchParams();

   useEffect(() => {
     const historyId = searchParams.get("historyId");
     if (!historyId) return;
     try {
       const raw = sessionStorage.getItem("polaris_restore");
       if (!raw) return;
       const saved = JSON.parse(raw);
       if (saved.id !== historyId || saved.page !== "peer") return;
       const result = saved.result as PeerCompareResult;
       setQuery(saved.query ?? "");
       setAId(result.a_ticker ?? "");
       setBId(result.b_ticker ?? "");
       if (result.fiscal_period) setFiscalPeriod(result.fiscal_period);
       setHasQueried(true);
       setPhase("done");
       setProgress(100);
       setPeerResult(result);
       sessionStorage.removeItem("polaris_restore");
     } catch {}
     // eslint-disable-next-line react-hooks/exhaustive-deps
   }, []);
   ```

   > 注意：peer 頁有「BQ 期別載入後自動把 `fiscalPeriod` 設成最新期」的 effect（約 510–525 行）。還原時被覆蓋的風險不大（還原的是 `peerResult` 而非靠 `fiscalPeriod` 重打 API），但驗收時請確認標題、KPI、比較摘要、tabs 都正確顯示。

4. **（選做）匿名 fallback `?q=`**
   - 未登入時 `handleItemClick` 會走 `router.push('/${item.page}?q=...')`。peer 頁目前也沒讀 `?q=`，所以匿名點 peer 紀錄一樣空白。
   - 可選擇：讀 `q` → 預填 `query`（要自動重跑查詢需公司能被解析，沿用現有 `runQuery` 邏輯即可）。
   - 主修先把登入者的 `historyId` 還原做好，這條可後續再補。
