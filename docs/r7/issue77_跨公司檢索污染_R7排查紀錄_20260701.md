# Issue #77 跨公司檢索污染 — R7 排查紀錄（2026-07-01）

## 背景

Issue #77（R4 於引用溯源核對時發現）：查「聯電2025Q3營收」，右側引用資訊面板前 7 筆全是別家公司（日月光、鴻海、聯詠），沒有一筆是聯電自己的資料。歸屬拆成兩塊：
- 檢索/實體解析（R3 主責）：缺硬性 ticker 過濾、聯電↔聯詠字形相近混淆
- badge doc_type→label 對應錯誤（R7 主責，前端顯示）

本篇記錄 R7 這邊的排查與修復，以及後續用台積電查詢重現同一類根因、比原始描述更嚴重的證據。

## R7 已修復（本次改動，皆已 commit）

### 1. citation `source_key` 對照表缺漏真實 BQ 值（commit `c31d712`）

直接查 `polaris_core.v_chunk_semantic` distinct `(doc_type, event_key, source_key)` 組合，只有 4 種正式值：

| doc_type | event_key | source_key |
|---|---|---|
| major_news | major_news.others | PRIMARY_MOPS |
| news | news | SECONDARY_NEWS_MEDIA |
| presentation | earnings_call | **PRIMARY_COMPANY_IR** |
| transcript | earnings_call | PRIMARY_EC_TRANSCRIPT |

前端 `frontend/src/lib/adapters.ts` 的 `_SOURCE_KEY_LABEL` 原本只有 `PRIMARY_EC_PRESENTATION` / `PRIMARY_NEWS` 這兩個真實資料裡從未出現的 key，導致 presentation/news chunk 的 badge 顯示原始英文 key（issue 裡「PRIMARY_COMPANY_IR」原字串直接出現在畫面上就是這個）。已補上 `PRIMARY_COMPANY_IR` / `SECONDARY_NEWS_MEDIA` 兩個真實 key 的中文對應，保留舊 key 不刪。

### 2. 研究查詢 race condition（commit `57d2b93`，對應 R6 報告）

R6 的 `R6_TSMC_2025Q1_Citation_Field_Check_2026-07-01.md` 報告：`/api/ask` 直測 4 次皆乾淨（全部台積電），但使用者在 UI 上看到聯發科/鴻海引用混入台積電查詢。排查發現 `research/page.tsx` 的 `run()` 沒有防重入：Enter 送出與建議 chip 都沒檢查 `running` 狀態（只有送出鈕/重新分析鈕有 `disabled`）。若使用者在上一題結果還沒回來前送出新問題，較慢的舊回應可能晚於新回應 resolve，蓋掉正確結果。已加 `if (phase === "running") return;` 單一擋點防護。

**此修復目前無法完全解釋使用者實測看到的污染**（見下方第 3 點的新發現）——race condition 是真實存在且已修復的 bug，但不是本次台積電案例污染的成因。

## 新發現：R6「API 乾淨」與使用者實測污染的落差原因

### 排查過程

本機環境設定修好後（`.env.local` 重複 key 蓋掉 `localhost:8000` 的問題，另記於 memory），使用者用前端「研究助理」頁面實測「台積電2025Q1法說會重點」，回報：摘要只顯示「根據台積電（233...」就截斷，引用資訊卻抓到聯詠、鴻海、聯發科。

繞過前端，直接對本機後端重打同一句查詢（`POST /research {"question": "台積電2025Q1法說會重點"}`），取得 raw JSON，`evidence` 陣列 8 筆：

| # | company | source_id | doc_type | fiscal_period |
|---|---|---|---|---|
| 0 | 鴻海 | `43c7d7e7-...`（UUID） | transcript | 2025Q4 |
| 1 | 聯詠 | `48e45c78-...`（UUID） | transcript | 2025Q4 |
| 2 | 鴻海 | `43c684bc-...`（UUID） | transcript | 2026Q1 |
| 3 | 鴻海 | `93767102-...`（UUID） | transcript | 2025Q1 |
| 4 | 鴻海 | `e2f78903-...`（UUID） | transcript | 2025Q1 |
| 5 | **台積電** | `2330-2025Q1-p011-c001` | presentation | 2025Q1 ✅ |
| 6 | 聯發科 | `2454-2025Q4-p011-c001` | presentation | 2025Q4 |
| 7 | 聯發科 | `2454-2025Q1-p003-c001` | presentation | 2025Q1 |

8 筆裡只有 1 筆是台積電；其餘 7 筆橫跨鴻海（4 筆）、聯詠（1 筆）、聯發科（2 筆），且期別也對不上（查 2025Q1，混進 2025Q4、2026Q1）。`final_answer` 內文甚至直接 narrate 了鴻海法人說明會邀請函原文（`ir@foxconn.com`、`www.honhai.com`）——污染在 writer agent 產出答案前就已存在於 evidence pool，前端只是忠實呈現後端給的資料，**這不是前端顯示問題**。

### 為何 R6 測不出來：端點不同

- R6 報告打的是 `/api/ask`（payload `{"query": ...}`）
- 前端「研究助理」頁面實際呼叫的是 `/api/research`（payload `{"question": ...}`，Deep Research ReAct 多步驟 agent），對應 `frontend/src/hooks/useResearch.ts`
- 查了 `frontend/src/hooks/useAsk.ts`（對應 `/ask`）：**整個 `frontend/src` 沒有任何頁面在用它**，是沒接到 UI 的孤兒 hook

R6 的 `/ask` 測試結果乾淨是真的，但不是使用者實際會走的路徑。`/research` 是多步驟 ReAct，每一步可能各自下不同的檢索查詢，比單次 RAG 的 `/ask` 更容易在某一步撈到無關公司。往後要複測這類污染問題，必須打 `/research`，用 `/ask` 驗證不出來。

## 結論 / 建議歸屬

1. **前端（R7）部分已修復完畢**：source_key 對照表補漏（badge label）、研究查詢防重入（race condition）。
2. **本次台積電案例的污染根因在 `/research` 端點的檢索層（R3）**，且比 issue #77 原描述的「聯電↔聯詠字形相近」更嚴重——混進完全不相關的鴻海、聯發科，期別也不對。應視為 issue #77 根因（缺硬性 ticker 過濾）的另一組具體重現證據。
3. 前端不打算加「只顯示命中查詢 ticker 的 citation」這類側邊過濾——治標不治本，還會刻意隱藏後端問題。
4. 建議 R3 之後複測一律用 `/research` + `{"question": ...}`，不要用 `/ask`，兩者結果不能互相代表。
