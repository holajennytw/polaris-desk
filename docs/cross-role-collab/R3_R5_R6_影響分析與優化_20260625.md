# R3 對 R5 / R6 的影響分析與團隊優化方向

> 整理日期：2026-06-25  
> 來源：[CTO 架構審查](R3_CTO_架構審查_20260625.md)、`src/polaris/eval/`、`src/polaris/ontology/`

---

## 一、現況問題總結

### R3 → R5 的依賴鏈

```
R5 出題（questions_v0.csv）
  ↓
runner.py 呼叫 build_workflow() / run_deep_research()
  ↓
R3 的 HybridRetriever.retrieve()
  ↓
contexts（找到的 chunks）餵給 Ragas
  ↓
Faithfulness / Context Precision / Answer Relevancy 分數
```

**R5 量到的品質 = R3 找到的資料的品質。**  
R3 找到假的，R5 就量假的。

### R3 → R6 的依賴鏈

```
R3 _result_to_citation()
  ↓ 呼叫 company_name(ticker)     ← R6 的 companies.py
  ↓ 使用 doc_type filter           ← R6 定義的分類值
  ↓ RESEARCH_DOC_TYPE_QUOTAS      ← hardcode，與 R6 ontology 沒有自動同步
```

## 二、R5 受影響的具體問題

### 問題 1：CI 跑出的 Ragas 分數沒有意義

題庫 `questions_v0.csv` 前幾題都問台積電（2330），剛好是 3 筆 stub 裡有的公司。所以：

- **CI（無 API key）環境**：BM25 找到 stub → `contexts = ["台積電 2025Q1 法說摘要：毛利率受到..."]` → Ragas 在量這句話的品質 → 分數可能還不錯，但代表的是「3 筆假資料的一致性」
- **真環境（有 key）**：vector search 找到真實 chunks → Ragas 才有意義

**結果**：CI 綠燈不代表真實品質沒問題。兩種環境的分數無法互相比較。

### 問題 2：smoke test 的 `contexts_nonempty` 會靜默通過

```python
# score.py
checks = {
    "contexts_nonempty": bool(record.contexts),  # 有 stub 就過
    "has_citations": record.citation_count >= 1,  # stub citation 也算
}
```

只要 BM25 stub 有東西，smoke 就過。但 stub 的內容跟問題可能完全不相關，這個 check 沒有偵測到品質退化。

### 問題 3：無法區分兩種失敗原因

R5 看到 `contexts = []` 或 `contexts = [stub]` 時，無法判斷是：
- A）R3 的 retriever 壞了（vector search 失敗）
- B）資料本來就沒入庫（R4 還沒跑 ingestion）

這讓 R5 無法正確歸因測試失敗的原因。

## 三、R6 受影響的具體問題

### 問題 1：`company_name()` hardcode 20 家，BQ 資料不同步

```python
# ontology/companies.py
_COMPANY_NAMES: dict[str, str] = {
    "2330": "台積電",
    "2317": "鴻海",
    # ... 20 家固定
}
```

R6 在 BQ `company_dim` 加入第 21 家公司時，R3 的 citation 裡 `company` 欄位仍是 `None`，前端顯示空白，但不會有任何 error。

### 問題 2：R6 的 ontology 資產沒有進 R3 的 filter

R6 建了 12 張 ontology 表（`industry_id`、`event_key`、公司分類⋯⋯），但 R3 的 filter 只用到三個最基礎的欄位：

| R3 現在用的 filter | R6 建好但 R3 沒用到 |
|---|---|
| `ticker`（公司代號） | `industry_id`（可做同業限縮） |
| `doc_type`（文件類型） | `event_key`（法說、月營收、重大訊息細分）|
| `fiscal_period`（季別） | `published_yyyymm`（已在 metadata，但沒進 filter UI）|

**影響**：使用者問「半導體產業法說重點」，R3 無法自動限縮到 `industry_id = 半導體` 的公司，只能靠 `detect_tickers()` 判斷有無提到特定公司名稱。

### 問題 3：`doc_type` 值沒有合約保護

R3 的 `RESEARCH_DOC_TYPE_QUOTAS` hardcode：

```python
# retriever.py
RESEARCH_DOC_TYPE_QUOTAS: Mapping[str, int] = {
    "transcript": 5,
    "major_news": 5,
    "news": 3,
}
```

R6 若修改 `doc_type` 的命名規則（例如 `transcript` 改成 `earnings_call`），R3 這裡靜默失效，搜尋空手而回，不報錯。

## 四、團隊優化任務

### R3 要做（主責）

**優先度 P0：修 BM25 corpus，讓 R5 eval 有意義**

在 app 啟動時從 BQ 載入真實 chunks 建 BM25 index，而非 hardcode 3 筆 stub。

```python
# 改善後概念（bm25_channel.py）
def _load_bm25_corpus(store: BigQueryStore) -> list[SearchResult]:
    """啟動時從 BQ 載入每家公司最新 2 季的 chunks（約 1 萬筆，100MB）。"""
    sql = """
    SELECT chunk_id, chunk_text, ticker, fiscal_period, doc_type
    FROM `polaris_core.chunks`
    WHERE fiscal_period IN (
      SELECT DISTINCT fiscal_period FROM `polaris_core.chunks`
      ORDER BY fiscal_period DESC LIMIT 2
    )
    """
    ...
```

**優先度 P1：`company_name()` 改為動態查 BQ**

```python
# 改善後概念
def company_name(ticker: str | None) -> str | None:
    # 優先查 BQ company_dim；查不到才 fallback 到本地 dict
    return _bq_company_name(ticker) or _COMPANY_NAMES.get(ticker)
```

**優先度 P2：在 response 加 `retrieval_channel_summary`**

讓 R5 能在 eval report 裡看到這次用的是哪些通道：

```json
{
  "final_answer": "...",
  "evidence": [...],
  "retrieval_channels_used": ["bm25", "embedding", "rerank"],
  "retrieval_degraded": false
}
```

---

### R5 要做（主責）

**優先度 P0：在 eval report 加 `retrieval_degraded` 欄位**

```python
# runner.py EvalRecord 加欄位
@dataclass
class EvalRecord:
    ...
    retrieval_degraded: bool = False   # 新增
    retrieval_channels: list[str] = field(default_factory=list)  # 新增
```

讓 report 可以分開顯示：
- `degraded=True` 的題目的分數（BM25 fallback，供參考）
- `degraded=False` 的題目的分數（真實 vector，才是有效分數）

**優先度 P1：題庫加入非台積電的題目**

現在 `questions_v0.csv` 前幾題全是台積電，BM25 stub 也有台積電，CI 分數虛高。
加入 stub 裡沒有的公司（例如聯電 2303、廣達 2382），讓 CI 能真正偵測到 `contexts = []` 的情況。

**優先度 P2：smoke test 加 `retrieval_degraded` 檢查**

```python
# score.py 新增 check
checks = {
    ...
    "not_fully_degraded": not record.retrieval_degraded,  # 新增
}
```

避免 BM25 stub 讓 smoke 靜默通過。

---

### R6 要做（主責）

**優先度 P0：確保 `doc_type` 值有合約文件**

建立 `docs/r6/ontology/doc_type_values.md`，列出所有有效的 `doc_type` 值及其意義，讓 R3 的 hardcode 有參考依據，並在改名前通知 R3。

```markdown
# doc_type 有效值（R6 維護）
| 值 | 意義 | 範例來源 |
|---|---|---|
| transcript | 法說會逐字稿 | MOPS 法說 PDF |
| major_news | 重大訊息 | MOPS 重大訊息 |
| news | 新聞 | 媒體新聞 |
```

**優先度 P1：與 R3 確認 `industry_id` filter 的接入方式**

評估讓 R3 在 filter 裡支援 `industry_id`，例如：

```python
# 使用者問「半導體產業」→ R3 filter 加 industry_id
filters = {"industry_id": "半導體", "doc_type": "transcript"}
```

這需要 R3 修改 `_FILTER_COLUMNS` 和 BQ SQL，R6 確認 `industry_id` 欄位在 `chunks` 表的對應方式。

**優先度 P2：公司新增時通知 R3 同步 `_COMPANY_NAMES`**

在 R3 改為動態查 BQ 之前，R6 每次新增公司到 `company_dim.csv` 時，同步更新 `ontology/companies.py` 的 `_COMPANY_NAMES`（`tests/test_company_names.py` 已有比對測試，可以 catch 不同步）。

## 五、優化執行順序與依賴關係

```
R3 修 BM25 corpus（P0）
  │
  ├──► R5 的 CI eval contexts 有真實資料 → Ragas 分數才有意義
  │
  └──► R5 可以拆分 degraded / non-degraded 分數（P0）

R3 加 retrieval_channels_used 到 response（P2）
  │
  └──► R5 的 EvalRecord 能記錄通道來源（P0）

R6 建 doc_type 合約文件（P0）
  │
  └──► R3 的 RESEARCH_DOC_TYPE_QUOTAS 有依據，不再是 magic string

R6 確認 industry_id 對應方式（P1）
  │
  └──► R3 接入 industry_id filter（P2）→ 語意搜尋品質提升
```
