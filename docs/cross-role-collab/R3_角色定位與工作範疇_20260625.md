# R3 角色定位與工作範疇

> Staff Engineer 分析｜2026-06-25  
> 適用版本：Polaris Desk 現有架構（BigQuery canonical `polaris_core`、LangGraph 5 節點、3-way HybridRetriever）

---

## 一、角色定位

### 最接近：RAG Engineer（主）＋ Backend Engineer（次）

| 職稱 | 符合度 | 說明 |
|---|---|---|
| **RAG Engineer** | ★★★★★ | 核心工作就是讓「找資料→餵給 LLM→產生有根據的回答」這條鏈路又快又準 |
| **Backend Engineer** | ★★★★☆ | 需要設計與維護 API layer，但重點不在 CRUD，而在把搜尋結果組裝成前端要的形狀 |
| AI Engineer | ★★★☆☆ | 有一部分，但 AI Engineer 通常還要做 prompt 設計與模型選型；這兩塊在本專案屬 R2 |
| Retrieval Engineer | ★★★☆☆ | 更專注於搜尋本身（Elasticsearch、Solr），本角色搜尋只是工具，最終目標是 RAG 品質 |
| Full Stack Engineer | ★☆☆☆☆ | 不碰 UI；前端由 R7 獨立維護 |

**為什麼是 RAG Engineer？**

本專案的核心難題不是「如何建 API」，而是：

1. **同一個問題，用 BM25 和 vector 找到的東西不一樣，哪個更對？** → Hybrid fusion + Rerank 邏輯
2. **LLM 生的答案有沒有根據？** → Citation metadata 是否正確對齊到原始 chunk
3. **找到的資料過期了嗎、錯公司了嗎？** → Metadata filter（`company`, `period`, `doc_type`, `viewer`）
4. **檢索壞掉時前端知道嗎？** → `retrieval_degraded` flag、fallback 路徑

這些問題的解法都在 Retriever Layer，不是一般 CRUD Backend。API layer 是「把 Retriever 的結果包裝好給前端」，是手段，不是目的。

## 二、責任範疇

### Must Have（必須負責）

- **HybridRetriever 實作與調優**：BM25 tokenization（中英混合 n-gram）、vector search（BigQuery VECTOR_SEARCH cosine）、RRF fusion score、Cohere rerank（`COHERE_API_KEY` opt-in）
- **Retriever 的 Metadata Filter**：`company`、`period`、`doc_type`、`viewer`（ACL）的 SQL 同步與 BM25 路徑一致性
- **Citation metadata 正規化**：確保所有檢索通道（BM25/stub/vector/rerank）都帶齊 `doc_type`、`published_at`、`fiscal_period`、`event_key`、`source_key`、`published_yyyymm`（`_CITATION_METADATA_KEYS`）
- **FastAPI endpoint 實作**：`POST /ask`、`POST /research`、`POST /contradiction`、`GET /chunk/{source_id}`、`POST /peer-compare`、`GET /alerts`、`GET /financials`、`GET /companies`、`GET /library`
- **`retrieval_degraded` 偵測**：判斷本次回應是否全來自 fallback（bm25/stub），回傳給前端顯示警告
- **API 與前端的欄位合約維護**：修改 response schema 前，先更新 `R3_需求清單_from_R7.md`
- **Fallback 路徑管理**：vector search 掛掉時確保 BM25 fallback 啟動，不讓前端白屏
- **`POST /contradiction` 矛盾偵測邏輯**：從 citations 中找數字不一致的段落

### Should Have（應該負責）

- **ColPali gated 整合**：`colpali_retriever.py` 在 round-trip 通過 ≥70% 後啟用進 prod（`scripts/colpali_roundtrip_check.py`）
- **Retriever 單元測試**：不呼叫真實 API（`GEMINI_API_KEY` 不存在時 BM25-only，CI 可重現）
- **`POST /resolve` 語意 ticker 識別**：能從「半導體龍頭」解析出 `2330`，取代前端字串比對
- **`GET /suggestions` 同業比較版（`?mode=peer`）**：動態生成提問 chip，非 hardcode
- **`POST /research` 補 `chart` + `kpis`**：workflow 跑完後，查 `financial_metrics` 附在 response

### Nice To Have（加分項）

- **Query rewrite**：使用者問「上季」→ 展開成當前季別再檢索
- **Citation dedup**：同一 source_id 多路命中時合併，不重複出現在 evidence
- **Latency profiling**：各通道耗時個別記錄（BM25/vector/rerank ms），供 eval 用
- **`POST /contradiction` 新格式**：從 `kpis+summary` 改接受 `citations[]`（見 `R3_需求清單_from_R7.md §2b`）

## 四、日常工作內容

**檢索品質優化（最頻繁）**
- 調整 BM25 + vector 的 RRF fusion 權重，跑 `eval/` 看 NDCG 變化
- 增加 metadata filter 欄位（如 `doc_type=transcript` 限縮法說題的雜訊）
- Rerank top-k 參數調整（目前 Cohere `top_n` 設定）

**API 維護**
- 新增 endpoint（如 `/resolve` 語意識別）或修改 response schema
- 修前端回報的欄位缺失問題（如 citation 缺 `fiscal_period`）
- 確認前端 mock 格式與真實 response 一致

**Citation 品質**
- 確認 BM25 路徑和 vector 路徑的 metadata key 對齊（`_CITATION_METADATA_KEYS`）
- debug `retrieval_degraded=true` 的根因（vector search 是否超時、BQ quota）

**矛盾偵測**
- 優化 `_has_provable_conflict()` 的數字萃取邏輯
- 測試不同 citation 組合的 false positive rate

**測試與可觀測性**
- 維護 retriever 單元測試（確保 CI 在無 API key 時仍跑通）
- 新增 logging 讓每次檢索的通道、耗時、命中數可追蹤

## 五、技能樹

### 必學

| 技能 | 原因 |
|---|---|
| Hybrid Search（BM25 + Dense Vector） | 核心架構；RRF fusion 是每天碰的東西 |
| Embedding 原理（cosine similarity、向量維度） | BigQuery VECTOR_SEARCH + gemini-embedding-2 768-dim |
| FastAPI（async、Pydantic schema、dependency injection） | API layer 全用 FastAPI |
| BigQuery SQL（VECTOR_SEARCH、JOIN、ARRAY<FLOAT64>） | 向量搜尋走 BQ，retriever 讀的是 BQ chunks |
| Python 3.13（dataclass、type hints、`from __future__ import annotations`） | 專案鎖 Python 3.13 |
| RAG 評估指標（Precision@k、Recall@k、NDCG、MRR） | 要知道怎麼量化「找得準不準」 |

### 重要

| 技能 | 原因 |
|---|---|
| Cohere Rerank API | opt-in 第 3 路；需知道 top_n 參數與品質 tradeoff |
| LangGraph 概念（節點、state、edge） | 要知道 workflow 怎麼呼叫 Retriever，否則無法對齊介面 |
| Metadata filter 設計（multi-field、ACL） | `company`/`period`/`doc_type`/`viewer` 四維過濾 |
| Chunk 策略（sliding window、fixed-size、semantic chunking） | 雖然 R4 做入庫，R3 需了解 chunk 結構才能寫正確 filter |
| Citation metadata 結構（`source_id`、`origin`、`event_key`）| 前端追溯引用的基礎 |
| uv / pyproject.toml | 專案套件管理工具 |

### 加分

| 技能 | 原因 |
|---|---|
| ColPali（視覺 PDF 檢索）| 第 4 路，gated，啟用後有明顯品質提升 |
| Ragas 框架 | 可跑自評估，不需等 R5 |
| Query expansion / HyDE | 提升 recall 的進階技巧 |
| Google Cloud（BQ IAM、VECTOR_SEARCH index 參數）| 了解底層有助於除錯 |
| Redis / cache 策略 | 高頻查詢快取，降低 BQ 費用 |
| OpenTelemetry trace | 分散式追蹤，跨 workflow→retriever 的 latency |

## 六、專案交付物

### 規格文件
- **API Spec**（`docs/cross-role-collab/R3_需求清單_from_R7.md`）：前後端合約，每個 endpoint 的 request/response schema
- **Retriever Architecture Doc**：4-way 檢索通道圖、fusion 策略、fallback 路徑、各通道啟用條件
- **Citation Metadata Schema**：每個 `origin` 值保證帶哪些 key、缺值時的預設行為

### 評估報告
- **Retriever Evaluation Report**：每次調整後的 Precision@k、Recall@k、MRR；與基準線比較
- **Search Benchmark**：100 題標準問題集 × 各 retrieval 通道的命中率（由 R5 出題，R3 執行）
- **Latency Report**：各通道 p50/p95 耗時，BQ vector search vs BM25 vs rerank

### 測試
- **Retriever Unit Tests**（`tests/test_retriever.py`）：無 API key 可跑、各通道獨立 mock
- **API Integration Tests**：endpoint 回傳 schema 正確性（Pydantic 驗証）

### 維運
- **Fallback Runbook**：vector search 掛掉時怎麼切 BM25-only mode，怎麼監控 `retrieval_degraded` 比例

## 七、KPI

### 技術指標

| 指標 | 目標 | 說明 |
|---|---|---|
| **Precision@5** | ≥ 0.70 | Top-5 citations 中有多少是真正相關的 |
| **Recall@10** | ≥ 0.80 | 所有相關 chunks 中有多少被找到 |
| **MRR（Mean Reciprocal Rank）** | ≥ 0.65 | 第一個相關結果平均排在第幾名 |
| **`retrieval_degraded` 比例** | < 10% | 全走 BM25/stub fallback 的查詢佔比；高表示 vector search 不穩 |
| **Retriever latency p95** | < 2s | BM25+vector（無 rerank）的 p95；rerank 可放寬至 4s |
| **Citation metadata 完整率** | 100% | 所有 citation 都帶齊 `_CITATION_METADATA_KEYS`；缺失會讓前端 DocViewer 壞掉 |
| **CI 通過率** | 100% | 無 API key 環境下 retriever tests 全綠 |

### 商業指標

| 指標 | 說明 |
|---|---|
| **查詢完成率** | 使用者送出查詢後，有多少次成功得到有 citation 的回答（非空回應）|
| **矛盾偵測 false positive 率** | 矛盾警示中有多少是誤報；過高會讓分析師不信任系統 |
| **`/financials` 資料覆蓋率** | 使用者查詢的 ticker 中，有多少能取得財務資料（衡量 R4 入庫完整性的反映指標）|

### 使用者體驗指標

| 指標 | 說明 |
|---|---|
| **Citation 可追溯率** | 前端點擊 citation 後能成功展開原文的比例（`GET /chunk` 成功率）|
| **降級警告曝光率** | `retrieval_degraded=true` 時前端正確顯示「備援資料」警告的比例；衡量使用者有無被告知資料品質 |
| **搜尋空結果率** | `evidence: []` 的查詢佔比；高表示檢索覆蓋不足，使用者拿到的是「AI 瞎編」的回答 |

## 附：R3 與鄰近角色的介面邊界

```
R4（資料工程）
  ↓ 寫入 polaris_core.chunks / financial_metrics / colpali_pages
  ↓ is_low_information() 過濾垃圾 chunk（2026-06-25 上線）
R3（本角色）
  ↓ 讀 BQ（VECTOR_SEARCH + JOIN）、BM25 in-memory、Cohere API
  ↓ 組裝 Citation[]、metadata normalization
R2（架構師）
  ↓ LangGraph workflow 呼叫 retriever.search()
  ↓ [visual_reader 節點] 讀圖 escalation（flag-gated，2026-06-25 加入）
  ↓ Writer node 拿 citations 填入 final_answer
R3（本角色）
  ↓ FastAPI response schema（含 citations、compliance_status）
  ↓ CitationOrigin：stub / bm25 / embedding / colpali / rerank / news / vision
R7（前端）
  ↓ adapters.ts 轉換成 ViewModel
  → 使用者看到引用接地的分析結果
```

- 不改 `frontend/src/`（屬 R7，前端有自己的 TypeScript 型別系統）
