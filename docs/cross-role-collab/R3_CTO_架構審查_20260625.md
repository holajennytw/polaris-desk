## 執行摘要

現行架構是一個 **MVP 品質的搭鐵線系統**：能跑、能 Demo、但有 6 個結構性缺陷，在資料量達到生產規模前就會觸發。最危險的是 BM25 fallback 從未真正索引 BigQuery 的資料——它搜的是 3 筆 hardcoded stub，這個設計在 10 家公司時就已是謊言，只是沒有人察覺。

## 一、現在的架構缺陷

### 缺陷 1：BM25 Fallback 是假的

**位置**：`retriever.py` `_FALLBACK_CORPUS`（第 57–82 行）、`_bm25_search()`

**問題**：BM25 搜尋的 corpus 是 **3 筆 hardcoded 字串**，跟 BigQuery 的真實資料完全無關。無論資料庫有多少 chunks，BM25 路徑只能找到這 3 筆。

```python
# retriever.py L57 - 這不是 fallback，這是裝飾品
_FALLBACK_CORPUS = [
    SearchResult(id="stub-2330-2025Q1-gm", content="台積電 2025Q1 法說摘要：..."),
    SearchResult(id="stub-2330-2024Q4-revenue", content="台積電 2024Q4 法說摘要：..."),
    SearchResult(id="stub-2317-2025Q1-segment", content="鴻海 2025Q1 法說摘要：..."),
]
```

**影響**：
- vector search 掛掉時，fallback 只回 2330 / 2317 的舊數據，完全不反映真實入庫狀況
- `retrieval_degraded` flag 應為「vector 失敗且 BM25 也沒有真實資料」，但目前永遠是假的準確

**風險等級**：🔴 Critical

---

### 缺陷 2：API Layer 沒有非同步

**位置**：`api.py` 所有 endpoint

**問題**：所有 endpoint 都是 `def`（同步），不是 `async def`。FastAPI 執行同步函式時會把它丟到 threadpool。在 `/ask` 和 `/research` 這種需要多次 BQ 查詢的端點，每個 request 都佔用一條 thread。

```python
# api.py L205 - 同步 blocking
def ask(req: AskRequest) -> AskResponse:
    result = build_workflow().invoke(...)  # 可能等 3–5 秒
```

**影響**：並發 20 個請求 → threadpool 滿 → 後續請求排隊 → 延遲雪崩

**風險等級**：🔴 High（並發量一上來立刻爆）

---

### 缺陷 3：HybridRetriever 每次 request 重建

**位置**：`api.py` `ask()` → `build_workflow()` → workflow invoke → `active_retriever()`

**問題**：每次 API 請求都建立新的 `HybridRetriever()` 實例，觸發 `get_vector_store()` → `BigQueryStore.__init__`，但 BQ client 是 lazy init，所以表面上沒有問題。

實際問題在 `BigQueryStore._get_client()`：每次新實例在第一次查詢時都呼叫 `bigquery.Client(project=...)`。BigQuery client 有初始化成本（credentials 載入 + gRPC 連線），重複建立是浪費。

**風險等級**：🟡 Medium（現在慢 50–100ms，高流量時明顯）

---

### 缺陷 4：VECTOR_SEARCH 的 WHERE 在子查詢內

**位置**：`bigquery_store.py` L124–141

```sql
FROM VECTOR_SEARCH(
    (SELECT * FROM `chunks` WHERE ticker = @ticker AND ...),
    'embedding', ...
)
```

**問題**：WHERE 在 VECTOR_SEARCH 的 base table subquery 內，BigQuery 的 ANN index 必須在過濾後的子集上重新計算。當 filter 很窄（例如只看一家公司的 5 個 chunks），ANN 退化成 exact scan，IVF index 完全失效。當 filter 很寬（無 filter），全表掃描成本 O(N)。

**影響**：
- 無 filter（問「整個市場怎麼看 AI」）→ 掃描全部 10M chunks
- 窄 filter（問「2330 最新一季」）→ ANN 效果差，可能漏掉相關 chunks

**風險等級**：🟡 Medium（現在 10 萬 chunks 勉強 OK，100 萬以上開始感受）

---

### 缺陷 5：矛盾偵測是純規則式，無法擴充

**位置**：`api.py` `_has_provable_conflict()`、`_numeric_tokens()`

**問題**：矛盾偵測用正規表達式抓數字，比對兩個 text 裡的數字是否有衝突。這個方法：
1. 無法偵測語意矛盾（「成長」vs「下滑」但沒有數字）
2. 無法跨 chunks 偵測（只比 kpi text vs summary text）
3. False positive 高：「營收成長 25%」vs「CoWoS 需求成長 25%」會誤判

**風險等級**：🟡 Medium（品質問題，不是崩潰問題）

---

### 缺陷 6：api.py 是上帝物件（937 行）

**位置**：`src/polaris/api.py`

**問題**：一個檔案包含：request model、response model、business logic（`_search_peer_calls`、`_metric_side`、`_peer_compare`）、endpoint、DB 查詢邏輯。`/peer-compare` 的業務邏輯（`_fmt_value`、`_metric_diff`）全在 api.py 裡，約 250 行。

**影響**：
- 兩個人同時改 api.py → git conflict 機率高
- 找一個 bug 要讀 937 行
- 無法對 peer-compare 的業務邏輯單獨測試

**風險等級**：🟡 Medium（維護成本問題）

## 七、10 家 → 1000 家 / 10 萬 → 1000 萬 chunks 崩潰分析

### 會在哪裡先崩？

**第一個崩：BM25 Fallback（現在就已經壞了）**

BM25 的 corpus 是 3 筆 hardcoded，與資料量完全無關。10 萬 chunks 或 1000 萬 chunks，BM25 的結果永遠是同樣 3 筆。這不是「規模問題」，這是「現在就是 bug」。

**第二個崩：BigQuery 費用（約 50 萬 chunks）**

| 資料量 | VECTOR_SEARCH 無 filter 每次費用 | 每日 100 查詢費用 |
|---|---|---|
| 10 萬 chunks | ~0.03 USD | ~3 USD |
| 100 萬 chunks | ~0.3 USD | ~30 USD |
| 1000 萬 chunks | ~3 USD | ~300 USD |

（估算基礎：每 chunk 768 float64 = 6.1 KB，BQ on-demand $5/TB）

無 cache、無 filter 的 VECTOR_SEARCH 每次掃全表。1000 萬 chunks × 768 dim = ~46 GB，每次查詢掃全表 = $0.23。100 用戶每天各問 10 次 = $230/day。**這個費用會在商業上先殺死系統。**

**第三個崩：並發（約 20 concurrent requests）**

同步 `def` endpoint + BQ 查詢 3–5 秒 + FastAPI threadpool 預設 40 threads。  
20 concurrent requests × 5s = threadpool 開始排隊 → p95 latency 從 5s 跳到 30s+。

**第四個崩：IVF Index 效率（約 500 萬 chunks）**

BigQuery VECTOR_SEARCH 在 WHERE 子查詢後使用 IVF index。當 chunks 到 500 萬，IVF 需要重建（BigQuery 不自動重建），否則 ANN 品質退化。重建一次要數小時，期間查詢走 exact scan。

---

### 崩潰優先順序

```
現在   → BM25 corpus 是假的（品質問題，不崩但回答爛）
~10萬  → 無明顯問題（現行架構的舒適區）
~50萬  → BQ 費用開始顯著，需要 cache + filter 強制
~100萬 → 並發壓力、IVF 需要規劃重建週期
~500萬 → IVF 必須重建，VECTOR_SEARCH latency 上升
~1000萬 → 整個 BigQuery on-demand 架構需要改成 slot reservation
```

## 八、改善方案與建議架構

### 短期（1 個月）：止血

**P0 — 修 BM25 Fallback**

從 BigQuery 定期（每日）匯出 chunks 的摘要版本（僅 `chunk_id`、`content`、`ticker`、`fiscal_period`），在 app 啟動時建立真實的 BM25 index。大小控制：取每家公司最新 2 季的 chunks（約 1 萬筆），記憶體佔用 ~100MB，可接受。

```python
# 改善後概念
class HybridRetriever:
    def __init__(self):
        self._bm25_corpus = load_bm25_corpus_from_bq()  # 啟動時載入，非 hardcode
        self._bm25_index = BM25Okapi([_token_list(c.content) for c in self._bm25_corpus])
```

**P1 — 加 Result Cache（Redis / BigQuery BI Engine）**

對 `(query_embedding_hash, filter_hash)` 做 cache，TTL 1 小時。同一查詢在 1 小時內不重打 BQ，直接回快取。

**P2 — api.py 拆分**

```
api.py → router/research.py（/ask, /research, /contradiction）
       → router/analytics.py（/peer-compare, /financials, /companies）
       → router/user.py（/history, /subscriptions）
       → services/peer_compare.py（從 api.py 抽出業務邏輯）
```

### 中期（2–3 個月）：架構穩固

**加 Request ID + Structured Logging**

```python
import uuid
@app.middleware("http")
async def add_request_id(request, call_next):
    request.state.request_id = str(uuid.uuid4())
    # 傳入 workflow、retriever、BQ query
```

**BQ Filter 最佳化**

將 WHERE 移到 VECTOR_SEARCH 外（post-filter），讓 ANN index 先找最相似的 top-k×3，再用 Python filter 過濾，避免 IVF 在小子集上退化。

**Contract Test（R3 ↔ R4）**

```python
# tests/contract/test_bq_schema.py
def test_chunks_columns_exist(bq_fake):
    required = {"chunk_id", "ticker", "doc_type", "fiscal_period", "embedding"}
    assert required.issubset(bq_fake.columns)
```

**Async Endpoints**

```python
@app.post("/ask")
async def ask(req: AskRequest) -> AskResponse:
    result = await asyncio.to_thread(build_workflow().invoke, {...})
```

### 長期（4–6 個月）：規模化

**VECTOR_SEARCH → Vertex AI Vector Search（若 chunks > 500 萬）**

BigQuery VECTOR_SEARCH 適合 100 萬以下。超過後考慮 Vertex AI Vector Search（Matching Engine），專門做 ANN，支援 horizontal scale 和 index 自動更新。BQ 繼續存 metadata，Vector Search 只存 embedding + id。

**BQ Slot Reservation**

1000 萬 chunks 時，on-demand 費用無法控制，改用 slot reservation（固定月費，不按掃描量計費）。
