## 方向一：Router 拆分

**難度**：低｜**建議時序**：本 Sprint

### 目標結構

```
src/polaris/
├── api.py              ← 只剩 app 建立 + middleware，~50 行
└── routers/
    ├── research.py     ← /ask, /research, /contradiction
    ├── analytics.py    ← /peer-compare, /financials, /companies, /library
    ├── user.py         ← /history, /subscriptions
    └── ops.py          ← /healthz, /alerts
```

### 改法

```python
# api.py（改後）
from polaris.routers import research, analytics, user, ops

app = FastAPI(...)
app.include_router(research.router)
app.include_router(analytics.router)
app.include_router(user.router)
app.include_router(ops.router)
```

### 好處

- 兩人同時開發不同功能，git conflict 機率降為零
- 找一個 endpoint 不需要讀 937 行
- 純機械搬移，不改任何邏輯，風險極低

## 方向二：Service Layer 抽象

**難度**：中｜**建議時序**：下 Sprint

### 目標結構

```
src/polaris/
└── services/
    ├── peer_compare.py     ← _search_peer_calls, _metric_side,
    │                          _fmt_value, _metric_diff（從 api.py 搬來）
```

### 好處

- 業務邏輯可以單獨測試，不需要起 FastAPI

- `/peer-compare` 的 250 行業務邏輯從 router 層消失，router 只剩「接請求、呼叫 service、回 response」

### 注意

搬移時同步補 unit test，確認行為不變再刪原始碼。

## 方向三：Retriever Channel 抽象

**難度**：較大｜**建議時序**：Q3，配合 BM25 修復一起做

### 目標結構

```
src/polaris/retrieval/
├── retriever.py          ← HybridRetriever 只負責 orchestration
├── channels/
│   ├── base.py           ← RetrievalChannel 抽象介面
│   ├── bm25_channel.py   ← BM25 邏輯（含真實 BQ corpus 載入）
│   ├── vector_channel.py ← BigQuery VECTOR_SEARCH
│   └── rerank_channel.py ← Cohere Rerank
└── fusion.py             ← RRF score 合併邏輯
```

### 介面設計概念

```python
# channels/base.py
class RetrievalChannel(Protocol):
    def search(self, query: str, filters: dict | None) -> list[SearchResult]: ...
    def is_available(self) -> bool: ...

# retriever.py（改後）
class HybridRetriever:
    channels: list[RetrievalChannel]  # 可注入任意通道組合

    def retrieve(self, query: str, *, filters=None) -> list[SearchResult]:
        candidates = []
        for ch in self.channels:
            if ch.is_available():
                candidates.extend(ch.search(query, filters))
        return self._rerank(_merge_results(candidates))
```

### 好處

- 加新通道（ColPali 正式啟用）只新增一個 class，不改主流程
- 每個通道可以獨立測試、獨立關閉（`is_available()` 回 False 即跳過）
- BM25 改成載入真實 BQ corpus 的邏輯只改 `bm25_channel.py`

## 對 R2 / R4 的影響

### R2（架構師）— 幾乎不影響

R2 只透過這兩個入口使用 Retriever：

```python
active_retriever()           # workflow Retriever 節點
make_research_search_fn()    # Deep Research
```

三個方向全部不動這兩個函式的簽名，R2 **零感知**。

**唯一需要確認的時機**：方向三執行時，若 `SearchResult` 欄位結構有變動，需與 R2 確認 workflow state 裡的 `Citation` dataclass 是否需要同步調整。改之前先發一條訊息確認即可。

### R4（資料工程）— 一個地方需打招呼

修 BM25 corpus（屬於方向三的一部分）時，R3 需要問 R4：

> 「app 啟動時想查一次 `polaris_core.chunks`，取每家公司最新 2 季的 chunks 建 BM25 index，這個讀取有沒有 quota 或 ACL 限制？」

R4 **不需要做任何改動**，只需確認這個 SELECT 查詢符合 `polaris_core` 的讀取規範（憲法 III：一般開發者有讀取權）。

其他方向（Router 拆、Service Layer）完全不碰 BQ schema，R4 **零影響**。
