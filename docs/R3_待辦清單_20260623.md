### 4. `GET /alerts` 補欄位 — 監控警示分頁

**症狀**：研究助理 + 同業比較的「監控系統警示」面板無法依頁面過濾。

**目前問題**：後端 `AlertResponse` 缺 `origin`、`title`、`source`、`time` 欄位；
前端已做 `normalizeAlert()` 轉換，但因 `origin` 為空，所有警示都流到兩頁。

**R3 需要加的欄位**：

| 欄位 | 型別 | 說明 |
|------|------|------|
| `origin` | `"research"` \| `"peer"` \| `"global"` | **必填**，前端路由用 |
| `title` | string | 一行標題（前端現在自行組「{ticker} · MOPS 監控」）|
| `source` | string | 來源描述 |
| `time` | string | `"HH:mm"` 或 ISO 字串 |

**前端配合動作**（R3 交付後 R7 執行）：
```ts
// research/page.tsx
const researchAlerts = [
  ...(alerts ?? []).filter(a => !a.origin || a.origin === "research"),
  ...contraAlerts
];
// peer/page.tsx
const peerAlerts = [
  ...(alerts ?? []).filter(a => !a.origin || a.origin === "peer"),
  ...contraAlerts
];
```

### 5. `POST /contradiction` — 矛盾偵測真實 LLM

**症狀**：矛盾偵測目前走 client-side 規則式 mock，非真實 LLM 偵測。

**Request**：
```json
{
  "kpis": [{ "label": "全年美元營收指引", "value": "中段 25%", ... }],
  "summary": [{ "text": "Q2 營收成長將達 25% 以上…", "cite": "...", "page": "p.7" }]
}
```

**Response（有矛盾時）**：
```json
{
  "alerts": [{
    "id": "contra-001",
    "origin": "contradiction",
    "level": "mid",
    "title": "全年指引表述不一致",
    "summary": "說明 + 建議",
    "source": "矛盾偵測 · …",
    "time": "14:22"
  }]
}
```

**Response（無矛盾時）**：`{ "alerts": [] }` → 前端自動顯示「交叉比對通過」（info level，已過濾不進通知中心）

## 架構備忘

### `GET /alerts` vs `POST /contradiction` 資料流

```
R3 Watchdog → GET /alerts
  ├─ origin="research" → 研究助理頁監控面板
  └─ origin="peer"     → 同業比較頁監控面板
  （前端 normalizeAlerts() 轉換；通知中心「風險動態」tab 亦顯示）

POST /contradiction → 前端 contraAlertStore（sessionStorage，page-scoped）
  ├─ level="high"|"mid" → 研究助理 / 同業比較頁監控面板顯示
  └─ level="info"       → 顯示「交叉比對通過」，不進通知中心
```

### 向量路修復（P1，#100）

- **根因確認**：`v_chunk_semantic` view 不含 `embedding` 欄，無法直接作為 VECTOR_SEARCH base table
- **R3 建議方向**：保留 `chunks` 做向量搜尋 → 用 `chunk_id` LEFT JOIN `v_chunk_semantic` 取得 `event_key`/`source_key`/`published_yyyymm`
- **R7 影響**：不需改程式碼；P1 合進後 citation 資料自動變為真實 embedding 資料
